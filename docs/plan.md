# 校园二手教材与物品交易平台最终计划

## 1. 文档定位

本文件是本项目首版最终实施计划，与 [goal.md](./goal.md) 保持一一对应。

使用规则：

1. 本文件回答“先做什么、做到什么算完成、哪些决定已经冻结”。
2. 首版范围、延期模块、AI 边界、图片规则已经冻结，不再反复摇摆。
3. 后续新增想法默认归入“后续扩展”，不直接改动首版合同。
4. 实际开发、测试、答辩准备都以本文件为准。

---

## 2. 已冻结的技术决策

### 2.1 运行环境

1. Java 版本固定为 `Java 8`。
2. 使用 `.envrc` 指向 `/usr/lib/jvm/java-8-openjdk`。
3. 构建工具固定为 `Maven`。
4. 图形界面固定为 `JavaFX`。
5. 数据库固定为 `SQLite`。
6. AI 能力固定为 `OpenAI Java SDK + 本地 Mock 兜底`。

### 2.2 架构决策

1. 采用 `MVC + Service + DAO` 分层结构。
2. 界面层只负责展示和事件转发，不直接操作数据库。
3. 业务规则集中放在 `Service` 层。
4. 数据读写集中放在 `DAO` 层。
5. 所有数据库访问统一经过 `DBUtil`。
6. 所有会话状态统一经过 `Session` 管理。

### 2.3 首版开发策略

1. 先完成主流程闭环，再补强后台与统计。
2. 先确保本地离线可演示，再接真实 AI。
3. 先保证功能稳定，再追求扩展数量。
4. 首版保留延期模块入口占位，但不实现完整业务流程。

---

## 3. 项目目标

本项目最终定位为：

> 一个基于 Java 8、JavaFX、SQLite 和 Maven 的校园二手教材与物品交易平台，支持普通用户注册登录、商品发布与浏览、商品图片本地管理、管理员审核、商品购买与订单记录、用户管理、操作日志、基础统计与图表展示，并以受控方式预留购物车、评价与更多 AI 能力扩展。

1. 创建 Maven 工程结构。

5. 引入 SQLite JDBC 依赖。

3. 数据库文件能自动生成。

## 8. 核心模块设计

### 8.1 app 层

职责：项目启动、窗口初始化、场景切换。

核心类：

1. `MainApp`
2. `SceneManager`

说明：

1. `MainApp` 负责启动 JavaFX。
2. `SceneManager` 统一处理页面跳转，避免 Controller 彼此直接创建窗口。

### 8.2 controller 层

职责：接收页面事件、调用 Service、更新界面。

首版需要的 Controller：

1. `LoginController`
2. `RegisterController`
3. `UserHomeController`
4. `GoodsDetailController`
5. `PublishGoodsController`
6. `UserCenterController`
7. `AdminHomeController`
8. `AdminGoodsReviewController`
9. `AdminUserController`
10. `AdminStatsController`
11. `AdminLogController`

约束：

1. Controller 不直接写 SQL。
2. Controller 不直接 new DAO。
3. Controller 只做输入校验、页面回显、调用服务。

### 8.3 service 层

职责：封装业务逻辑与规则。

首版需要的 Service：

1. `UserService`
2. `GoodsService`
3. `OrderService`
4. `StatisticsService`
5. `AdminLogService`
6. `AiService`
7. `ImageStorageService`

关键业务规则：

1. 注册时用户名唯一。
2. 被封禁用户不能登录。
3. 商品发布后默认待审核。
4. 只有审核通过且在售商品可被购买。
5. 购买行为必须原子化处理订单创建和商品售出状态更新。
6. 已售商品不能再次购买。
7. 卖家不能购买自己的商品。
8. 卖家只能下架自己的未售商品。
9. 管理员封禁、解封、审核、删除违规商品都必须写日志。
10. 商品图片支持 `0-3` 张本地保存与主图选择。

### 8.4 dao 层

职责：负责数据库 CRUD。

首版需要的 DAO：

1. `UserDao`
2. `GoodsDao`
3. `GoodsImageDao`
4. `OrderDao`
5. `AdminLogDao`
6. `StatisticsDao`

说明：

1. DAO 只处理数据库访问，不处理业务判断。
2. 查询条件尽量参数化，避免拼接 SQL。

### 8.5 util 层

职责：放置公共基础工具。

首版需要的工具类：

1. `DBUtil`
2. `DateUtil`
3. `AlertUtil`
4. `Session`
5. `PasswordUtil`
6. `AppConfig`

说明：

1. `PasswordUtil` 负责密码哈希，不允许明文存储。
2. `AppConfig` 负责读取环境变量和默认配置。
3. `Session` 负责保存当前登录用户。

## 9. 数据库设计基线

### 9.1 首版必须落地的表

2. `goods`
3. `goods_images`
4. `orders`
5. `reviews`
6. `admin_logs`

### 9.3 数据设计原则

1. 所有主键统一使用 `INTEGER PRIMARY KEY AUTOINCREMENT`。
2. 所有时间字段统一保存为文本时间戳。
3. 状态字段统一用明确枚举值。

7. 已售商品不能重复购买。
8. 购买后订单生成且商品状态变为 `SOLD`。
9. 下架只能作用于自己的未售商品。
10. 用户管理聚合字段返回发布数和成交数。
11. 统计字段覆盖首版要求的基础数字和分类统计。
12. 图片规则覆盖 `0` 张、`1` 张、`3` 张、超限失败、主图选择与无图占位。
13. AI 在真实不可用时回退 Mock。

## 16. 风险与应对

### 16.1 Java 8 兼容风险

风险：

1. 新版依赖可能默认要求更高 JDK。

应对：

1. 所有依赖选择明确兼容 Java 8 的版本。
2. 编译目标固定为 `1.8`。

### 16.2 AI 网络风险

风险：

1. 答辩现场可能无网络。
2. OpenAI 调用可能超时或失败。

应对：

1. 保留 Mock 回退。
2. UI 上将 AI 定位为辅助功能，不绑定主流程。

### 16.3 功能过载风险

风险：

1. 延期模块过早实现会冲击主流程稳定性。

应对：

1. 严格按首版范围推进。
2. 延期模块只保留入口占位和结构预留。

### 16.4 数据为空导致统计页难看

风险：

1. 演示数据不足时图表和表格缺乏说服力。

应对：

1. 提前准备多类别商品和至少一条订单数据。
2. 准备有图和无图商品，保证图片逻辑可演示。

### 16.5 文档与实现漂移风险

风险：

1. 种子数据、测试数据、页面选项可能偏离固定分类枚举。
2. 文档要求的统计页和日志页可能晚于主流程实现。

应对：

1. 把分类枚举约束写入测试。
2. 把日志页、用户管理聚合和统计字段列为首版硬验收项。

## 17. 最终结论

本项目首版不再追求“看起来什么都有”，而是追求：

1. 主流程完整。
2. 后台能力可答辩。
3. 图片、日志、统计不留模糊地带。
4. AI 亮点可演示但不绑死主流程。
5. 延期模块边界清楚，后续扩展有落点。

只要按本计划完成并通过测试，首版即可视为达到课程设计交付标准。
