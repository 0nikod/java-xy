# 校园二手教材与物品交易平台最终计划

## 1. 文档定位

本文件是本项目的最终实施计划，不再作为可随意扩写的需求草稿使用。

使用规则：

1. 本文件优先回答“先做什么、做到什么算完成、哪些决定已经冻结”。
2. 除非出现课程硬性要求变化或技术约束变化，否则不轻易修改本文件。
3. 后续新增想法默认归入“后续扩展”，不直接改动首版目标。
4. 实际开发、测试、答辩准备都以本文件为准。

---

## 2. 已冻结的技术决策

以下决策视为当前项目基线，不再反复讨论：

### 2.1 运行环境

1. Java 版本固定为 `Java 8`。
2. 使用 `.envrc` 指向 `/usr/lib/jvm/java-8-openjdk`。
3. 构建工具固定为 `Maven`。
4. 图形界面固定为 `JavaFX`。
5. 数据库固定为 `SQLite`。
6. AI 能力固定为 `OpenAI Java SDK + 本地 Mock 兜底`。

### 2.2 架构决策

1. 采用 `MVC + Service + DAO` 分层结构。
2. 界面层只负责展示和事件转发，不直接操作数据库。
3. 业务规则集中放在 `Service` 层。
4. 数据读写集中放在 `DAO` 层。
5. 所有数据库访问统一经过 `DBUtil`。
6. 所有会话状态统一经过 `Session` 管理。

### 2.3 首版开发策略

1. 先完成主流程闭环，再做扩展功能。
2. 先确保本地离线可演示，再接真实 AI。
3. 先保证功能稳定，再追求功能数量。
4. 首版不追求一次做完全部模块。

## 3. 项目目标

项目目标不是简单堆功能，而是交付一个可答辩、可演示、可维护的 Java 课程设计项目。

本项目最终定位为：

> 一个基于 Java 8、JavaFX、SQLite 和 Maven 的校园二手教材与物品交易平台，支持普通用户注册登录、商品发布与浏览、管理员审核、商品购买与订单记录、基础统计与图表展示，并预留购物车、收藏、留言、举报、评价与 AI 智能分析扩展能力。

1. 新用户可以注册并登录。

2. Maven 命令可成功执行。

## 6. 开发阶段计划

后续开发顺序固定如下，默认不打乱。

### 第一阶段：项目基础搭建

目标：把空目录变成可运行的 Java 8 Maven JavaFX 项目。

必须完成：

1. 创建 Maven 工程结构。
2. 编写 `.envrc`。
3. 配置 `JAVA_HOME`、`PATH`。
4. 编写 `pom.xml`。
5. 引入 SQLite JDBC 依赖。
6. 引入 OpenAI Java SDK 依赖。
7. 创建 `MainApp` 启动类。
8. 创建基础 FXML 和样式文件。
9. 创建数据库初始化脚本。
10. 准备演示种子数据。

阶段完成标准：

1. `mvn clean compile` 可以通过。
2. JavaFX 主窗口可以启动。
3. 数据库文件能自动生成。

### 第二阶段：用户与商品主流程

目标：打通最核心交易链路。

必须完成：

1. 注册页。
2. 登录页。
3. 用户首页。
4. 商品发布页。
5. 商品详情页。
6. 商品搜索、分类、排序。
7. 管理员审核页。
8. 单商品购买。
9. 订单生成。

阶段完成标准：

1. 普通用户发布商品。
2. 管理员审核通过。
3. 另一用户购买。
4. 订单在数据库中落地。

### 第三阶段：个人中心与后台管理

目标：把项目从“能买卖”提升到“能管理、能展示”。

必须完成：

1. 个人中心。
2. 我的发布。
3. 下架商品。
4. 我的购买订单。
5. 我的卖出订单。
6. 用户管理页。
7. 封禁与解封。
8. 操作日志。
9. 基础统计。

阶段完成标准：

1. 管理员能管理用户和商品状态。
2. 前后台都能回看业务结果。
3. 图表页可稳定展示演示数据。

目标：给项目增加亮点，但不影响主流程稳定性。

必须完成：

1. 商品描述优化。
2. 管理员运营摘要。

1. 有 Key 时可以真实调用。

3. 不因 AI 失败阻断主流程。

## 8. 核心模块设计

### 8.1 app 层

职责：项目启动、窗口初始化、场景切换。

核心类：

1. `MainApp`
2. `SceneManager`

说明：

1. `MainApp` 负责启动 JavaFX。
2. `SceneManager` 统一处理页面跳转，避免 Controller 彼此直接创建窗口。

### 8.2 controller 层

职责：接收页面事件、调用 Service、更新界面。

首版需要的 Controller：

1. `LoginController`
2. `RegisterController`
3. `UserHomeController`
4. `GoodsDetailController`
5. `PublishGoodsController`
6. `UserCenterController`
7. `AdminHomeController`
8. `AdminGoodsReviewController`
9. `AdminUserController`
10. `AdminStatsController`

约束：

1. Controller 不直接写 SQL。
2. Controller 不直接 new DAO。
3. Controller 只做输入校验、页面回显、调用服务。

### 8.3 service 层

职责：封装业务逻辑与规则。

首版需要的 Service：

1. `UserService`
2. `GoodsService`
3. `OrderService`
4. `StatisticsService`
5. `AdminLogService`
6. `AiService`

关键业务规则：

1. 注册时用户名唯一。
2. 被封禁用户不能登录。
3. 商品发布后默认待审核。
4. 只有审核通过且在售商品可被购买。
5. 购买行为必须原子化处理订单创建和商品售出状态更新。
6. 已售出商品不能再次购买。
7. 卖家不能购买自己的商品。
8. 卖家只能下架自己的未售商品。
9. 管理员封禁、解封、审核、驳回都必须写日志。

### 8.4 dao 层

职责：负责数据库 CRUD。

首版需要的 DAO：

1. `UserDao`
2. `GoodsDao`
3. `OrderDao`
4. `AdminLogDao`
5. `StatisticsDao`

说明：

1. DAO 只处理数据库访问，不处理业务判断。
2. 查询条件尽量参数化，避免拼接 SQL。

### 8.5 util 层

职责：放置公共基础工具。

首版需要的工具类：

1. `DBUtil`
2. `DateUtil`
3. `AlertUtil`
4. `Session`
5. `PasswordUtil`
6. `AppConfig`

说明：

1. `PasswordUtil` 负责密码哈希，不允许明文存储。
2. `AppConfig` 负责读取环境变量和默认配置。
3. `Session` 负责保存当前登录用户。

## 11. AI 设计基线

AI 是亮点，不是主流程依赖。

### 11.1 首版保留的 AI 功能

1. 商品描述优化。
2. 管理员运营摘要。

### 11.2 首版不纳入的 AI 功能

1. 定价建议。
2. 购买建议。
3. 违规风险分析。
4. 图表解读细分能力。

### 11.3 调用策略

1. 默认支持真实 OpenAI SDK 调用。
2. 如果未配置 `OPENAI_API_KEY`，自动切换到 Mock。
3. 如果真实调用失败，自动回退 Mock。
4. AI 返回结果只作为建议，不参与业务判定。

### 11.4 配置原则

1. API Key 通过环境变量读取。
2. 不写死在源码。
3. `.envrc` 只负责本地开发环境，不提交真实密钥。

## 12. 环境配置基线

### 12.1 `.envrc`

计划内容：

```bash
export JAVA_HOME=/usr/lib/jvm/java-8-openjdk
export PATH="$JAVA_HOME/bin:$PATH"
```

如果后续需要本地 AI 调试，可以额外配置：

```bash
export OPENAI_API_KEY=your_key_here
```

注意：

1. 示例值可以写入文档，但不提交真实 Key。
2. 团队开发时应使用各自本地环境变量。

### 12.2 Maven 基线

`pom.xml` 至少需要满足：

1. `maven.compiler.source=1.8`
2. `maven.compiler.target=1.8`
3. SQLite JDBC 依赖
4. OpenAI Java SDK 依赖
5. 测试依赖

### 12.3 JavaFX 基线

因为目标环境是 Java 8，默认依赖 JDK 8 自带 JavaFX。

这意味着：

1. 首版尽量不引入需要新 JDK 的 JavaFX 生态组件。
2. 页面和样式保持简洁，优先保证兼容性。

## 16. 风险与应对

### 16.1 Java 8 兼容风险

风险：

1. 某些新库默认偏向更高 JDK。

应对：

1. 依赖选型优先确认 Java 8 支持。
2. 不引入非必要高级框架。

### 16.2 AI 网络风险

风险：

1. 答辩现场可能没有网络。

应对：

1. 必须有 Mock。
2. 演示时不把 AI 当成唯一亮点。

### 16.3 功能过载风险

风险：

1. 功能太多导致主流程不稳。

应对：

1. 严格按本计划的首版范围执行。
2. 未进入首版范围的功能不插队。

### 16.4 数据为空导致统计页难看

风险：

1. 没有样例数据时，图表页和统计页无法展示价值。

应对：

1. 强制准备种子数据。
2. 初始化脚本内置演示数据。

---

## 17. 最终结论

本项目后续开发统一遵守以下顺序：

1. 先搭环境和骨架。
2. 再打通注册、登录、发布、审核、购买、订单主线。
3. 再补个人中心、用户管理、统计和图表。
4. 最后接 AI，并始终保留 Mock 兜底。

本计划的核心原则只有一句话：

> 先交付稳定、完整、可答辩的主流程，再做扩展和亮点，不为了功能数量破坏首版完成度。
